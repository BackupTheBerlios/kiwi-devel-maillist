> Yes, but intentionally did not use the functionality of the statefull
> XML object. I am still hoping to have a stateless XML object (I thought
> you were in agreement with that goal). The way __checkFilesystemTool()
> was implemented it was independent of state inside the XML object. I am
> hoping over time to decrease the code that is dependent on state in the

I knew you would say that :) I only used the getImageTypeAndAttributes()
from the KIWIXML object which accesses the data structures of the
XML object and not more. getImageTypeAndAttributes() itself doesn't
create a state. If we change KIWIXML to be completely stateless this
function will still exist and if not it have to have a successor because
I use it in several other places to obtain exactly the same
information you need in __checkFilesystemTool() too.

imho it's not a good idea to write additional functions in the 
processing code only used by the runtime checker and also available
elsewhere.

If the XML object is stateless and if that action changes some
functions or their meaning it's better to find the same accessor
code in all places and change it accordingly at that time

new functions also means new maintenance overhead and a potential
risk for even more bugs. And that for the reason of a change which
we did not have implemented so far.

Hope you don't mind but I found this not to be necessary to
that extend
