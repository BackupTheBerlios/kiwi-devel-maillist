Hi,

> I don't think the current implementation will produce the desired
> result. If for example a user defines
> 
> <profile name="a"/>
> <profile name="b"/>
> 
> <preferences profiles="a">
> <type image="vmx" format="vmdk".../>
> ..
> </preferences>
> <preferences profiles="b">
> <type image="vmx" format="qcow2".../>
> ..
> </preferences>
> 
> Then runs kiwi with --add-profiles a,b

What would be the desired result in such a case. Beside the fact
that it's a nonsense setup the result will be that only one can
win :) At the moment we don't print any warning message in that
case which we should but at the end only one of both vmx types
can be build. Currently the first one selected (vmdk) is the winner
and imho that's ok for such a weird profile setup 

> Anyway, it goes back to state. If we have no state and just store the
> data then for the XML object we do not have to jump through mental hoops
> to understand/figure out what it actually stores. The answer is simple.
> It stores everything in config.xml.

This is correct but as soon as you come to the state which you have
to at some point you have to make the same decisions. So what I'm
trying to say is a stateless XML object doesn't solve the problem
it shifts it to another level but at the end one out of two data
sets here is not used at all even if present in the XML object 

But I agree it would be a clean and better implementation if
data is one level and state decision based on data is
another level
