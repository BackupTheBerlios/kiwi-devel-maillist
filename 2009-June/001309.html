> I think a better solution for what I want to do would be a new kiwi format 
> that just dumped the root filesystem out to an image, and a new boot method 
> (pxe, usb and iso) that replaced the existing root with the new one. 
> 
> Sadly though, I don't think my level of perl skills extend to figuring
> out how to do that. 

Hmm, yes it needs some coding. Maybe you think of using kiwi just for
the creation of the root filesystem image. If you specify something 
like:

  <type>ext3</type>

kiwi only creates the filesystem blob. You also can create the
initrd with kiwi in a separate call just by using one of the
provided boot image descriptions in /usr/share/kiwi/image/*boot.
The result at the end would be:

- the filesystem blob plus md5 sum
- the initrd cpio file
- the kernel file

You can start coding in your preferred language to bring that
data on a device as you wish. The boot code inside the initrd
is plain bash coding which I think can be easily adapted to your
needs.

> Although, saying that, there is probably a use case for the OEM image to be 
> able to only extract the root partition in a system rebuild situation. 

The oem image type supports a recovery mode which we use for the
preoload images sent out to HP and Lenovo. This mode just restores from
a backup the files from first deployment. So the oem system recovery
is already in place
