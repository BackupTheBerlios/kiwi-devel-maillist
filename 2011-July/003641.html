Hi,

> The kiwi-tools package that is built along with the core "kiwi"
> package doesn't have "/lib/mkinitrd/run-init" executable.
> The kiwi-tools package distributed rhel-repo directory has the
> run-init executable. 

*urks*

> What would be a good way to fix this issue? 
> Options are:
> i) Add the run-init exec code to the base packages and make the executable also part of kiwi-tools base package.
> ii) Enable yum priorities support for kiwi and make sure the right package is picked up while building the boot image. 
> iii) Of course there is the easy fix of adding the required execs to the archive(root) directory of the isoboot/rhel-05.4 boot configuration.

I vote for ii). Is it possible to specify a full qualified name in yum
like "kiwi-tools-4.22-85.1" so that we can be sure this one is picked
up ? It works that way in smart and zypper

> Starting from RHEL6 on, there is not going to be any more mkinitrd support in the base distribution. So is the plan to adopt any of the dracut components or keep re-compiling the mkinitrd for RHEL6 as well? I have to start working on RHEL/CentOS 6 soon, so wanted to align my efforts with that of the upstream.

This is an interesting information, so how does RHEL6 create the initrd
for re-booting the system ? The 'mkinitrd' call happens in preinit so
we are already in the system at that point. I'm fine with adding the RHEL6
way here too. It would be nice if the code could be compatible to RHEL5
though

> Finally the kiwi Gitorious (at least the web interface) 
> is not accessible :(. Was the repo moved to elsewhere?

yes that's true. I wrote about that in the last weekly report. It has
moved to github here:
