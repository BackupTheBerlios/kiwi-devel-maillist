Hi,

User point of view:

With the commit 0cc08153a27779c62384338803ee4483301f41f1 the boot attribute of 
the type tag in config.xml supports either relative paths that are mapped to 
wherever kiwi is installed or absolute paths pointing wherever you want.

Relative paths on the one hand are usually considered relative to the location 
they are used in (here the directory of the config.xml) but that is not the 
case with kiwi and thus imho confusing.
If one wants to use content version systems absolute paths pointing to custom
boot image config.xml will on the other hand cause trouble if checked out on a 
different machine.

That is why this patch provides support for kiwi's URL magic to the boot
attribute.
One can now use this:// to point to a path relative to the config directory.
Due to consistency this patch also adds support for system:// which can be
used to point to wherever the images are installed that are shipped with kiwi 
(that was originally achieved by a relative path but would be confusing 
compared to the behaviour of this://).

Developer point of view:

Marcus complained about default:// so I renamed it to system:// (which is not 
perfect but hopefully better).
At the moment only local directories are supported (this://, obs://, 
system://, dir:// should work). I only tested this:// and system://. I 
personally see no reason why samba shares should not be supported, as kiwi 
mounts the shares they can be treated as local directories.

Could someone please review this patch?


Regards,
Franz
